# v0.31+ Roadmap — post-design-rewrite

Written at the end of v0.30 series. v0.30.0 → v0.30.3 finished the
**structural** UI rewrite (home, focus mode, complete-checkin, profile,
calendar) per `docs/design-handoff/`. The app's visual shell is now
"true to the handoff" rather than "old layout with new colors."

This doc captures what's next, grouped by intent.

---

## 1. Finish the design migration (small, mechanical)

These are the screens v0.30 did **not** restructure. None are blockers
for daily use — most users won't see them often — but they remain
visually inconsistent with the rest of the app.

| Screen | Current state | Action |
|---|---|---|
| Achievement wall (`package-profile/badges`) | Uses v0.25 hero + grouped grid; cat photos are bundled (✓) but the page chrome predates the design tokens. | Restructure per `screen-achievements.jsx` — rarity-grouped sections with the cat photo as the primary glyph. |
| Six-subject progress (`package-profile/subjects`) | Compact list with bars; works but doesn't match design language. | Token migration (no structural change required). |
| Settings (`package-profile/settings`) | Sectioned form with v0.25 chrome. | Token migration. |
| Daily report poster (`package-profile/share`) | Image-export surface; visual style is intentionally distinct. | Audit but don't restructure — sharing surface should keep its own identity. |
| Public profile (`package-profile/public-profile`) | Read-only roll-up. | Token migration. |
| Settings sub-page: 公开学习页 | Single toggle row inside settings. | Token migration. |
| Cold-start quote modal | Already on new tokens (v0.29). | No action. |
| Weekly/monthly recap modals | Use legacy `--mint-600`/`--text-main` tokens. Those tokens are still defined in `app.wxss` so they render fine; just stale. | Optional cleanup pass — low priority. |

**Estimated effort:** 1–2 ship cycles. Each is a contained WXSS-only
patch; no server changes, no test surface changes.

## 2. Quality-of-life (focus-on-focus)

The product's stated axis is 专注 + CPA 备考. After the visual rewrite,
these are the highest-leverage product moves before adding new
surfaces:

### 2a. Pause behavior + auto-end
- Current: pause is unbounded — a user who walks away and forgets
  ends up with a session that drifts. Server stores the pause
  duration, so we can detect this server-side.
- Action: if a session has been paused for >30 minutes, auto-complete
  with the elapsed run-time (not run+pause). UX: notification +
  toast on next app open ("上次有一段没结束的专注，已经按 N 分钟自动完成").
- Server: extend `/api/sessions/:id/auto-complete` (new endpoint) or
  fold into existing complete with `auto: true` flag.

### 2b. Background timer protection
- Current: closing the miniprogram while a session is running keeps
  the server clock ticking but client state is rebuilt from `/me`
  on cold start (good). However, if the user kills the process mid-
  session and never returns, the session sits "running" forever in
  the DB.
- Action: server-side cron sweep (already have a 20:30 reminder cron)
  to find sessions older than 8 hours and auto-complete them.

### 2c. Focus-mode mid-session subject pick
- Current: subject is chosen at complete time only (intentional —
  the v0.21 URL-encoding bug taught us not to round-trip through
  query params).
- Action: optional mid-session subject pick via a small tap target
  in focus mode — locally cached, applied at complete time. The
  user can change their mind. No URL involvement, so the v0.21 bug
  class can't recur.

### 2d. Streak-recovery polish
- Current: makeup-banner shows when server flags a recoverable
  streak. The mechanism works but the banner currently sits above
  the exam hero, fighting the user's intended scan path.
- Action: move the banner inside the timer card (as a small inline
  notification strip) when relevant, so the eye sees exam → timer →
  recovery → goal in one column rather than five top-level blocks.

## 3. CPA 备考 features (study-specific)

After v0.30.3 the UI is calm enough to host new study-specific
features without overwhelming the user. Two candidates worth
exploring:

### 3a. Per-subject target minutes + automatic split
- The user has 6 subjects with very different weights in the CPA
  exam. Currently subjects compete for the same daily goal.
- Action: weekly/per-subject targets. When a session ends with a
  subject pick, count toward that subject's weekly target. Surface
  delta in the six-subject grid card.
- Server: `subject_targets` table (user_id, subject, weekly_minutes).
  Already have subject in session rows.

### 3b. Topic-level tagging within a subject
- "会计 · 金融资产" is a different topic than "会计 · 长期股权投资".
  Users currently can't distinguish.
- Action: free-text topic on the complete page (small input under
  subject). Build a "recent topics" auto-complete from prior
  sessions. No server schema change — store in the existing summary
  field with a parseable prefix, OR add a `topic` column.

### 3c. Mock-exam mode (longer-horizon)
- Distinct mode from pomodoro: 180-minute timed session that
  doesn't allow pause. Useful as the exam approaches.
- Action: extend session schema to support `mode: 'mock-exam'`
  alongside the existing `focus` / `pomodoro`. Pause button hidden
  in focus mode for this mode; only "结束" is available.

## 4. Reliability + analytics

Maintenance work that pays dividends but doesn't ship visible features.

### 4a. Client-side error reporting
- Currently `console.error` calls go to nowhere on user devices.
- Action: lightweight error reporter that POSTs a single anonymized
  event per error. Server stores last 100 per user (rolling). View
  via an admin endpoint guarded by `cpa-admin-token`.

### 4b. Real-device QA helper
- The ui-review-checklist exists but is checked manually. Worth
  building a tiny "design QA" page (admin-only) that surfaces every
  v0.30-class problem we've shipped: font scales, tap targets,
  emoji/CJK pairs, dark-mode contrast samples. One-stop check before
  ship.

### 4c. Migration safety net
- Sessions in the DB still carry v0.18-era encoded subject strings
  in rare edge cases. Server has parsing fallbacks but they're
  defensive code that nobody trusts. Worth a one-time backfill pass
  + remove the fallbacks.

## 5. What we explicitly NOT-do

Things that have been considered and rejected; recorded here so future
sessions don't re-litigate.

- **No "social feed" or public timeline.** The user explicitly axed
  "动态" in v0.22. Don't reintroduce it as a different name.
- **No second tab for achievements.** The achievement wall is one
  click from profile; it doesn't earn a tab slot.
- **No screen-on lock by default.** v0.22.1 made setKeepScreenOn
  default OFF after the user noticed battery drain. Don't flip the
  default back without a clear reason.
- **No URL-encoded subject hand-offs.** v0.21.2 burned us; pick
  subject on complete page only.
- **No system-following theme for focus mode.** The focus mode is
  always dark by design — it's a sensory shift, not a theme. Don't
  add a setting to switch this off (would dilute the "this is a
  different surface" feeling).

## 6. Sequencing

If asked "what next" with no further input, the order I'd take:

1. **v0.30.4** (mechanical):
   - Achievement wall token migration
   - Subjects + settings + public profile token migration
2. **v0.31.0** (small but meaningful):
   - Pause auto-end (server cron + client toast)
   - Mid-session subject tap target
3. **v0.31.1**:
   - Per-subject weekly target + grid refactor
4. **v0.32.0** (heavier):
   - Mock-exam mode
5. **v0.32.1**:
   - Topic-level tagging
6. **v0.33+** (maintenance):
   - Error reporter
   - DB backfill + fallback removal

Each row is a separate, shippable patch. v0.30.3 verification gates
all of this — confirm the design rewrite renders correctly on the
user's device before adding new product surface.
